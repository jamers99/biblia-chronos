# Data Schema & Import Spec

A spec for the data layer: the canonical schema we own, and the importer that
turns the Theographic snapshot into it. Written to be implemented directly. See
[ARCHITECTURE.md](../ARCHITECTURE.md) for the reasoning behind these choices.

## The two stages

There are two transformations, and they must stay separate:

1. **Import** (`import:theographic`) — Theographic raw export → our canonical
   seed files. Run occasionally, by hand, reviewing the diff. This spec covers
   this stage.
2. **Build** — canonical files → small per-visualization JSON bundles shipped to
   the browser. Runs every build. Out of scope here, but the schema below is
   what it consumes.

The importer regenerates `data/seed/` wholesale, so **seed files are never
hand-edited**. Human corrections live in `data/overrides/` and are layered on top
of seed at build time (overrides win, matched by `id`). This is what lets us
re-import without clobbering contributor fixes. The merge itself is a build-stage
concern; the importer only writes `data/seed/`.

## Directory layout

```
data/
  vendor/theographic/      # pinned snapshot of the upstream json/ folder
    SNAPSHOT.md            # human-readable: source URL, commit SHA, license, counts
    snapshot.json         # machine-readable: pinned commit + committed/fetched file lists
    books.json chapters.json events.json people.json peopleGroups.json places.json   # committed
    verses.json easton.json   # gitignored — fetched via vendor:fetch
  seed/                    # GENERATED by the importer — do not hand-edit
    people.json places.json events.json peopleGroups.json books.json edges.json
  overrides/               # hand-authored corrections/additions (importer never touches)
  sources.json             # hand-authored source table (see Sources)
scripts/
  vendor-fetch.mjs         # restores the gitignored vendor files (already implemented)
  import-theographic.ts    # the importer
src/lib/data/
  schema.ts                # Zod schemas (single source of truth for types)
```

**What's tracked in git:** `seed/`, `overrides/`, `sources.json`, and most of
`vendor/` are committed; the build output (shipped bundles) is gitignored.
Committing `seed/` is what makes a re-import a reviewable diff in a PR — which is
the whole point of the determinism rule below — and keeps the build (and GitHub
Pages deploy) a pure file read rather than something that must run the importer.

The two heaviest vendor files are **not** committed: `verses.json` (~37MB, needed
only at import time to resolve verse refs) and `easton.json` (~6MB, unused by the
current importer). They're fetched on demand from the pinned commit by
`scripts/vendor-fetch.mjs` (`npm run vendor:fetch`), driven by
`data/vendor/theographic/snapshot.json`. So the seed data lives twice (raw
`vendor/` plus transformed `seed/`), but the heavy raw files don't bloat the repo,
and `seed/` remains the durable canonical copy regardless.

## Canonical schema

All schemas live in `src/lib/data/schema.ts` as Zod, with types derived via
`z.infer`. The importer validates its output against these before writing.

### Shared conventions

- `id` — a stable, human-readable, URL-safe slug (e.g. `aaron_1`). This is our
  canonical key. Airtable `rec…` ids are used only as a lookup *during* import
  and never appear in output.
- `source` — a key into `sources.json` (e.g. `"theographic"`). Required on every
  entity and edge. Keeps provenance to one short string per record.
- `status` — `"published" | "draft"`. Drafts stay in seed (so edges remain
  valid); the build decides whether to ship them.
- Years are numbers; negative means BCE, using the upstream convention as-is
  (e.g. `-1574`). Document, don't reinterpret.

### Value types

```ts
const Year = z.number().int()

const DateEstimate = z.object({
  earliest: Year.optional(),
  latest:   Year.optional(),
  display:  z.string().optional(),     // "c. 1574 BC"
  chronology: z.string().optional(),   // source key for the chronology used
  source:   z.string().optional(),     // overrides the record's source for this date
})

const Coordinates = z.object({ lat: z.number(), lng: z.number() })

const ScriptureRef = z.object({
  book:     z.string(),                // book id, e.g. "gen"
  chapter:  z.number().int().positive(),
  verse:    z.number().int().positive(),
  verseEnd: z.number().int().positive().optional(),
})
```

### Entities

```ts
const Base = z.object({
  id: z.string(),
  name: z.string(),
  status: z.enum(["published", "draft"]).default("published"),
  source: z.string(),
  description: z.string().optional(),
  scriptureRefs: z.array(ScriptureRef).optional(),
})

const Person = Base.extend({
  kind: z.literal("person"),
  displayTitle: z.string().optional(),
  isProperName: z.boolean().optional(),
  gender: z.enum(["male", "female", "unknown"]).optional(),
  birth: DateEstimate.optional(),
  death: DateEstimate.optional(),
  lived: DateEstimate.optional(),      // overall min/max span
})

const Place = Base.extend({
  kind: z.literal("place"),
  coordinates: Coordinates.optional(),
  featureType: z.string().optional(),      // "City", "Water", "Region"
  featureSubType: z.string().optional(),   // "River"
  modernName: z.string().optional(),
  altNames: z.array(z.string()).optional(),
})

const Event = Base.extend({
  kind: z.literal("event"),
  date: DateEstimate.optional(),           // start..end (end derived from duration)
})

const PeopleGroup = Base.extend({
  kind: z.literal("peopleGroup"),
})

const Book = z.object({                    // reference table, not a Base entity
  id: z.string(),                          // osisName lowercased, e.g. "gen"
  name: z.string(),                        // "Genesis"
  shortName: z.string().optional(),        // "Ge"
  osis: z.string(),                        // "Gen"
  order: z.number().int(),                 // bookOrder
  chapterCount: z.number().int(),
  testament: z.enum(["Old Testament", "New Testament"]),
  division: z.string().optional(),         // bookDiv, "Pentateuch"
})

const Source = z.object({
  id: z.string(),
  title: z.string(),
  author: z.string().optional(),
  url: z.string().url().optional(),
  license: z.string().optional(),          // "CC-BY-SA-4.0"
  kind: z.enum(["dataset", "book", "dictionary", "scholar", "other"]).optional(),
  notes: z.string().optional(),
})
```

### Edges

Relationships are generic edges, not fields on entities. The runtime builds the
reverse direction itself, so store each fact once.

```ts
const EdgeType = z.enum([
  "parent-of",        // parent -> child
  "sibling-of",       // symmetric
  "half-sibling-of",  // symmetric
  "partner-of",       // symmetric
  "member-of",        // person -> peopleGroup
  "participated-in",  // person -> event
  "occurred-at",      // event  -> place
  "died-at",          // person -> place
  "visited",          // person -> place
])

const Edge = z.object({
  from: z.string(),
  to: z.string(),
  type: EdgeType,
  source: z.string(),
})
```

Rules:
- **Directional** edges (`parent-of`, `member-of`, `participated-in`,
  `occurred-at`, `died-at`, `visited`) are stored in the listed direction only.
- **Symmetric** edges (`sibling-of`, `half-sibling-of`, `partner-of`) are stored
  once with endpoints ordered so `from < to`, to dedupe automatically.
- Drop any edge whose endpoint doesn't resolve to a known entity id, and count it
  in the report. Do **not** silently keep dangling edges.

## Theographic → canonical mapping

Every upstream record is wrapped as `{ id, createdTime, fields: {…}, modified }`.
Read the real data from `.fields`, the Airtable rec-id from `.id`, and **drop
`createdTime`/`modified`** (they'd churn diffs). Field names below are from the
current snapshot — the implementer should open the vendored files and confirm,
mapping any additional useful fields and ignoring the rest.

### Slugs (canonical ids)

| Entity | Upstream slug | Strategy |
|---|---|---|
| Person | `personLookup` (`aaron_1`) | use as-is |
| Place | `placeLookup` (`abana_1`) / `slug` | use as-is |
| Book | — | lowercase `osisName` → `gen` |
| Event | none | `slugify(title)`, collisions broken by appending `eventID` |
| PeopleGroup | none | `slugify(groupName)`, collisions broken by a stable upstream id |

Generated slugs must be **stable across re-imports**: derive from a stable field
(title/groupName) and break ties with a stable upstream id (e.g. `eventID`), not
a running counter.

### Per-entity field mapping

**Person** (`people.json`): `name`←name, `displayTitle`←displayTitle,
`isProperName`←isProperName, `gender`←lowercase gender, `description`←`dictText`
(array → joined, markup stripped), `status`←status. Dates: `birth` from
`birthYear`, `death` from `deathYear`, `lived` from `minYear`/`maxYear` (parse
numeric-string years; a single known year sets both `earliest` and `latest`).
Relationship fields (`father`, `mother`, `children`, `siblings`,
`halfSiblingsSameMother`, `halfSiblingsSameFather`, `partners`, `memberOf`) →
edges (see below). `verses` → resolve to `scriptureRefs`.

**Place** (`places.json`): `name`←`esvName` (fallback `kjvName`/slug),
`featureType`, `featureSubType`, `status`. Coordinates: prefer
`latitude`/`longitude`, then `openBibleLat`/`openBibleLong`, then
`recogitoLat`/`recogitoLon`; skip if none. `kjvName`/`esvName` → `altNames`.
`modernName` is **not** populated by the import — it's left empty for an override
to fill in. `verses` → `scriptureRefs`. Relationship fields `peopleDied`,
`hasBeenHere`, `eventsHere` → edges.

**Event** (`events.json`): `name`←`title`, `status` defaults `published` (no
status field upstream). `date`: `earliest` from `startDate`; `latest` from
`startDate` + `duration` (`"930Y"`→years, `"7D"`→days≈same year, `"1Y"`→1 year —
parse the trailing unit). Keep `sortKey` only if the build needs it (otherwise
drop). `participants` → `participated-in` edges, `locations` → `occurred-at`
edges, `verses` → `scriptureRefs`.

**PeopleGroup** (`peopleGroups.json`): `name`←`groupName`, `status` defaults
`published`. `members` → `member-of` edges (person → group).

**Book** (`books.json`): `bookName`, `osisName`, `shortName`, `bookOrder`,
`chapterCount`, `testament`, `bookDiv` → the Book fields above.

**Verses** (`verses.json`, ~10MB): not shipped as entities. Load it only to build
a map of verse `rec-id` → `ScriptureRef` (book id + chapter + verse), used to
resolve every `verses` array on other records. Confirm the exact book/chapter/
verse field names in the file.

### Relationship → edge map

| Source field | Edge | Notes |
|---|---|---|
| `father`, `mother` | `parent-of` (parent→child) | |
| `children` | `parent-of` (this→child) | dedupe against father/mother |
| `siblings` | `sibling-of` | symmetric, order endpoints |
| `halfSiblingsSameMother/Father` | `half-sibling-of` | symmetric |
| `partners` | `partner-of` | symmetric |
| `memberOf`, peopleGroups `members` | `member-of` | dedupe across both sources |
| events `participants` | `participated-in` | |
| events `locations`, places `eventsHere` | `occurred-at` | dedupe across both |
| places `peopleDied` | `died-at` | |
| places `hasBeenHere` | `visited` | |

All endpoints are upstream rec-ids → resolve to canonical ids via the per-type
lookup; drop + count any that don't resolve.

## Import algorithm

1. Read the vendored snapshot from `data/vendor/theographic/`.
2. Build per-type maps of rec-id → canonical id (assign/generate slugs first).
3. Build the Book table and the verse rec-id → `ScriptureRef` map.
4. Transform each entity: read `.fields`, map fields, parse dates, resolve
   `scriptureRefs`. Keep drafts (`status: "draft"`).
5. Build edges from relationship fields; resolve endpoints; normalize symmetric
   edges; dedupe; drop + count danglers.
6. Validate all output against the Zod schemas; abort on failure.
7. Write `data/seed/*.json`, arrays sorted by `id` (edges sorted by
   `from,to,type`).
8. Write an import report: counts per entity/edge type, dropped edges, unresolved
   refs, generated-slug collisions.

## Determinism

Re-running the importer on the same snapshot must produce byte-identical output
(a clean `git diff`). That means: drop `createdTime`/`modified`, sort every output
array by a stable key, generate slugs deterministically, and use no timestamps or
randomness.

## Sources table

`data/sources.json` is hand-authored (not generated) and starts with one entry
the importer references by default:

```json
[{
  "id": "theographic",
  "title": "Theographic Bible Metadata",
  "author": "Robert Rouse",
  "url": "https://github.com/robertrouse/theographic-bible-metadata",
  "license": "CC-BY-SA-4.0",
  "kind": "dataset"
}]
```

The importer sets `source: "theographic"` on everything it produces and should
fail if that id is missing from `sources.json`.

## Tooling

- TypeScript, run with `tsx`: `npm run import:theographic`.
- Dependency footprint: `zod` (+ `tsx` dev). Plain `node:fs` for IO.
- The importer reads only from `data/vendor/`, never the network. The two
  gitignored vendor files must be present first, so the full re-import is
  `npm run vendor:fetch && npm run import:theographic`. `vendor:fetch`
  (`scripts/vendor-fetch.mjs`, already implemented, no deps) restores them from
  the pinned commit in `snapshot.json`.

## Acceptance criteria

- `npm run import:theographic` runs clean and all output validates against the
  Zod schemas.
- Output is deterministic: running it twice yields no `git diff`.
- No dangling edges; every `source`/`scriptureRefs.book` resolves.
- The report prints non-zero counts for people, places, events, groups, and each
  edge type.
- Spot checks pass, e.g. Aaron (`aaron_1`) has a `parent-of` edge from Amram, a
  `sibling-of` edge with Moses, and resolved scripture refs.

## Settled decisions

1. **Drafts in seed:** upstream `wip` records are imported as `status: "draft"`
   so edges stay valid; the build decides per-visualization whether to show them.
2. **Verses/chapters as entities:** not yet. We ship a Book table plus refs
   resolved to `{ book, chapter, verse }`. Full verse/chapter entities are
   deferred until a visualization needs them.
3. **Periods & Kingdoms:** not imported. They're hand-authored under
   `data/overrides/` when the first timeline needs them.
4. **Place names:** display `name` comes from `esvName` (fallback `kjvName`, then
   slug), with both kept in `altNames`. `modernName` is left empty by the import
   and filled in via overrides.
